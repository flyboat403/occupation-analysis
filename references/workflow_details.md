## 步骤1：检索专业教学标准

### 数据源优先级

```
┌─────────────────────────────────────────────────────────────┐
│  1. 从 moe_pdfs_final.json 查找专业                         │
│     ↓                                                       │
│  2. 找到？                                                   │
│     ├─ YES → 获取 PDF URL                                   │
│     │         ↓                                             │
│     │    有 PDF URL？                                        │
│     │    ├─ YES → 下载并解析PDF → 返回完整内容             │
│     │    └─ NO  → 返回基本信息（无PDF URL）               │
│     │                                                       │
│  3. 未找到？ → 从 moe_pdfs_md 目录解析MD文件 🔄             │
└─────────────────────────────────────────────────────────────┘
```

### 数据源说明

| 优先级 | 数据源 | 内容 | 状态 |
|--------|--------|------|------|
| 1 | `moe_pdfs_final.json` | 专业代码、名称、PDF URL | 可用 |
| 2 | PDF文档解析 | 完整专业教学标准内容 | 可用（需安装pdfplumber或pypdf）|
| 3 | `moe_pdfs_md/*.md` | 已解析的MD文档 | 可用（备用）|

### 执行命令

```bash
# 完整检索（包含PDF解析）
python scripts/search_major.py --major "专业名称" --level "教育层次" --output temp/major_info.json

# 仅获取基本信息（不解析PDF）
python scripts/search_major.py --major "专业名称" --level "教育层次" --no-pdf --output temp/major_info.json
```

### 输出字段

| 字段 | 说明 | 来源 |
|------|------|------|
| `major_code` | 专业代码 | 必需 |
| `major_name` | 专业名称 | 必需 |
| `education_level` | 教育层次 | 必需 |
| `pdf_url` | PDF文档链接 | moe_pdfs_final.json |
| `pdf_parsed` | PDF是否已解析 | 布尔值 |
| `career_orientation` | 职业面向 | PDF解析或MD文件 |
| `training_goal` | 培养目标 | PDF解析或MD文件 |
| `main_courses` | 主要课程 | PDF解析（可选）|
| `data_source` | 数据来源 | primary/fallback |

## 步骤2：获取职业信息（大模型处理）

### ⚠️ CRITICAL - 必须使用 OccupationDictionaryLoader

> **本步骤必须调用 `OccupationDictionaryLoader` 加载职业大典数据。**
> 
> **禁止行为**：直接读取职业信息而不查询职业大典，会导致职业定义缺失。
>
> **正确做法**：
> 1. 根据职业代码首位确定大类文件
> 2. 使用完整职业代码在文件中检索
> 3. 提取职业定义和主要工作任务

### OccupationDictionaryLoader 使用方法

```python
# 在大模型处理步骤中，必须调用 OccupationDictionaryLoader
from scripts.occupation_dict_loader import OccupationDictionaryLoader

loader = OccupationDictionaryLoader()

# 根据职业代码加载对应大类
# 例如：职业代码 6-05-01-01 → 加载 class_6_生产制造及有关人员.md
occupation_data = loader.load_by_occupation_code('6-05-01-01')

# 输出包含：
# - code: 职业代码
# - name: 职业名称
# - definition: 职业定义
# - main_tasks: 主要工作任务列表
```

### 加载职业大典文档

| 职业代码首位 | 大类文件 | 内容示例 |
|-------------|----------|----------|
| 1 | `class_1_党的机关...负责人.md` | 党政机关负责人 |
| 2 | `class_2_专业技术人员.md` | 工程技术人员、设计人员 |
| 3 | `class_3_办事人员...md` | 行政办公人员 |
| 4 | `class_4_社会生产服务...md` | 服务人员 |
| 5 | `class_5_农林牧渔...md` | 农业生产人员 |
| 6 | `class_6_生产制造...md` | 制造业人员（服装制版师、缝纫工等） |
| 7 | `class_7_2025年新增职业.md` | 新职业 |

### 输出格式

```json
{
  "occupations": [
    {
      "code": "4-12-02-01",
      "name": "汽车机械维修工",
      "definition": "使用工、夹、量具和仪器仪表，检修设备，维护、修理和调试汽车发动机、底盘等机械部件的人员。",
      "tasks": ["汽车发动机检修", "汽车底盘检修", "传动系统维护"],
      "source": "职业大典2022"
    }
  ],
  "jobs": [
    {"name": "汽车维修", "related_occupation": "4-12-02-01"}
  ]
}
```

## 步骤3：利用大模型世界知识推断ESCO/O*NET映射

### 核心思路

> **本步驟由执行本Skill的Agent利用自身大模型的世界知识完成推断。**
> 
> **原理**：
> - 大模型训练数据包含公开的ESCO（欧盟职业分类）和O*NET（美国职业分类）体系
> - 无需本地文档依赖，直接利用大模型内置知识推断
> - 后处理验证编码格式确保输出正确
> 
> **优势**：
> - 无外部依赖（网络、本地文档）
> - 处理新职业能力强（大模型知识覆盖）
> - 实现简单（仅需提示词模板）

### 推断流程

```
┌─────────────────────────────────────────────────────────────────────┐
│  Step 3: 大模型世界知识推断ESCO/O*NET映射                             │
│                                                                      │
│  1. 输入：occupation_dict_data.json                                  │
│     ├─ 职业编码：2-02-38-02                                           │
│     ├─ 职业名称：物联网工程技术人员                                    │
│     └─ 职业定义：从事物联网架构、平台、芯片、传感器...                 │
│                                                                      │
│  2. Agent构造提示词                                                  │
│     ├─ 输入：职业名称+编码+定义                                       │
│     ├─ 提示词模板：职业分类专家角色设定                                │
│     └─ 要求输出：ESCO编码+名称、O*NET编码+名称                        │
│                                                                      │
│  3. 大模型利用世界知识推断                                           │
│     ├─ 分析职业定义核心能力                                          │
│     ├─ 匹配ESCO职业（欧盟职业分类体系）                               │
│     ├─ 匹配O*NET职业（美国职业分类体系）                              │
│     └─ 输出编码+名称+置信度+推断依据                                  │
│                                                                      │
│  4. 后处理验证                                                       │
│     ├─ ESCO编码格式：4位纯数字                                        │

│     └─ 必需字段完整性检查                                            │
│                                                                      │
│  5. 输出：occupation_mapping_info.json                               │
│     ├─ esco_code: 2152                                               │

│     ├─ onet_code: 17-2072.00                                         │
│     ├─ onet_name: Electronics Engineers, Except Computer             │

│     └─ mapping_reason: "物联网涉及电子设备、传感器，与电子工程师匹配" │
└─────────────────────────────────────────────────────────────────────┘
```

### ⚠️ CRITICAL - 编码格式强制规范

> **必须严格遵守以下编码格式，格式错误会导致文档查询失败**

| 数据源 | 编码格式 | 示例 | 文件命名规则 |
|--------|----------|------|--------------|
| **ESCO** | 4位数字 | `2636`、`7231`、`7541` | `{编码}.1.md`（如 `2636.1.md`） |
| **O*NET** | XX-XXXX.XX | `27-1022.00`、`49-3023.00` | `{编码}.md`（如 `27-1022.00.md`） |

**常见错误示例**：
| 错误写法 | 正确写法 | 说明 |
|----------|----------|------|
| `ISCO-2636` | `2636` | ESCO不需要ISCO-前缀 |
| `27-1022` | `27-1022.00` | O*NET必须带.00后缀 |
| `2636.00` | `2636` | ESCO不需要小数后缀 |

**输出验证**：生成mapping后必须检查：
```python
# ESCO编码验证：必须是4位纯数字
assert len(esco_code) == 4 and esco_code.isdigit()

# O*NET编码验证：格式必须是 XX-XXXX.XX
assert re.match(r'^\d{2}-\d{4}\.\d{2}$', onet_code)
```

## 步骤5：整合基础数据

> **MANDATORY - 必须使用 `scripts/integrate_data.py`**
> 
> 本步骤由 Python 脚本执行，不做语义推断，仅做数据搬运。

### 执行命令

```bash
python scripts/integrate_data.py \
  --major temp/major_info.json \
  --occupation temp/occupation_dict_data.json \
  --mapping temp/occupation_mapping_info.json \
  --output temp/combined_data.md
```

### 整合内容（4个数据源）

| 数据源 | 文件 | 输出Markdown标题 |
|--------|------|-----------------|
| 专业教学标准 | `major_info.json` | `# 专业教学标准` |
| 中国职业信息 | `occupation_dict_data.json` | `# 职业信息` |
| 国际职业代码映射 | `occupation_mapping_info.json` | `# 国际职业代码映射` |
| ESCO/O*NET原始文档 | 按映射代码直接从 `assets/` 读取 | 按职业分节 |

### 输出格式

```markdown
---
生成时间: 2026-05-01 10:00:00
数据格式: Markdown
---

# 专业教学标准
- 专业代码：740202
- 专业名称：西餐烹饪
...

# 职业信息
## 职业1：西式烹调师
- 职业代码：4-03-02-03
...

# 国际职业代码映射
| 中国职业 | ESCO代码 | ESCO名称 | O*NET代码 | O*NET名称 |

# ESCO/O*NET 原始文档
## 职业1：西式烹调师
```

### 关键约束

- **零提取、零转换**：直接读取原始MD文件内容，不做解析转换
- **按需读取**：根据映射中的ESC代码读取 `assets/esco_details_md/`，O*NET代码读取 `assets/onet_details_md/`
- Python脚本只做数据搬运，不做语义推断

## 边缘场景详细处理

### 步骤1边缘场景：PDF解析失败

**诊断检查**：

```bash
# 检查PDF URL是否有效
curl -I "https://example.com/path/to.pdf"

# 检查本地缓存
ls -la temp/pdf_cache/

# 检查备选数据源
ls assets/moe_pdfs_md/ | grep "专业代码"
```

**Fallback 方案**：

| 场景 | 命令 |
|------|------|
| PDF下载失败 | `python scripts/search_major.py --major "专业名称" --no-pdf` |
| 使用备选MD文档 | Agent直接读取 `assets/moe_pdfs_md/` 对应文件 |

### 步骤2边缘场景：职业大典格式异常

**常见异常及修复**：

```markdown
1. 职业代码格式错误
   - 错误示例：`4120201`（缺少分隔符）
   - 正确格式：`4-12-02-01`
   - 修复：手动补充分隔符

2. 职业定义缺失
   - 处理：从职业名称推断定义
   - 或从相近职业复制定义模板

3. 任务列表为空
   - 处理：从ESCO/O*NET对应职业反推任务
   - 示例：若ESCO有10个任务，翻译为中文后使用
```

**验证脚本**：

```python
# 验证职业信息完整性
import json
with open('temp/occupation_info.json', 'r', encoding='utf-8') as f:
    data = json.load(f)
    
for occ in data.get('occupations', []):
    issues = []
    if not occ.get('code'):
        issues.append("缺少职业代码")
    if not occ.get('definition'):
        issues.append("缺少职业定义")
    if not occ.get('tasks'):
        issues.append("缺少任务列表")
    
    if issues:
        print(f"[WARNING] {occ.get('name', '未知职业')}: {', '.join(issues)}")
```

### 步骤3边缘场景：映射低置信度处理

**低置信度判断标准**：

| 因素 | 权重 | 评分标准 |
|------|------|----------|
| 职业定义相似度 | 40% | 高度相似=40，部分相似=20，差异大=0 |
| 任务匹配度 | 40% | >70%=40，50-70%=20，<50%=0 |
| 行业领域一致性 | 20% | 同行业=20，相关行业=10，不同行业=0 |

**置信度计算**：
- 总分≥60 → `high`
- 总分40-59 → `medium`
- 总分<40 → `low`

**低置信度处理模板**：

```
【映射置信度警告】

职业名称：{中文名称}
职业代码：{中国代码}

推断结果：
- ESCO：{英文名称} ({代码}) - 置信度: {low/medium}
- O*NET：{英文名称} ({代码}) - 置信度: {low/medium}

原因分析：{具体差异说明}

建议操作：
1. 确认 - 接受当前映射继续
2. 修改 - 手动指定ESCO/O*NET代码
3. 跳过 - 不使用国际数据，仅用中国职业大典

请选择：
```

### 文档缺失处理

**缺失处理策略**：

```
文档缺失判断逻辑：
├─ ESCO文档存在 且 O*NET文档存在 → 正常使用
├─ 仅ESCO文档存在 → 使用ESCO + 职业大典任务
├─ 仅O*NET文档存在 → 使用O*NET + 职业大典任务
└─ 两者都缺失 → 仅使用中国职业大典，报告标注

报告标注格式：
> **数据完整性说明**：部分职业的国际参考数据（ESCO/O*NET）缺失，
> 本报告基于中国职业分类大典（2022版）编制。
```

**用户确认面板**：

```
【文档缺失警告】

以下职业的国际文档缺失：

┌──────────────────────────────────────────────────────────────────┐
│ 职业名称           │ ESCO文档 │ O*NET文档 │ 影响程度           │
├──────────────────────────────────────────────────────────────────┤
│ 新能源汽车维修工   │ ⚠️ 部分  │ ✅ 存在   │ 技能分析可能不完整 │
│ 汽车美容师         │ ❌ 缺失  │ ❌ 缺失   │ 仅使用中国职业大典 │
└──────────────────────────────────────────────────────────────────┘

影响说明：
- ESCO文档缺失：技能和知识描述可能不完整
- O*NET文档缺失：工作任务和能力描述可能不完整
- 两者都缺失：将仅使用中国职业大典数据

是否继续？
- 回复"确认"继续（使用已有数据）
- 回复"修改"调整职业映射
- 回复"取消"终止本次分析
```

**自动恢复机制**：

```
错误类型判断：
├─ 可恢复错误（网络超时、格式异常）
│   └─ 自动尝试Fallback方案
├─ 需用户确认错误（文档缺失、映射低置信度）
│   └─ 展示警告面板，等待用户决策
└─ 不可恢复错误（关键数据文件丢失）
    └─ 终止流程，报告错误原因和修复建议
```
